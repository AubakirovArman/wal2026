## Question

Can old edits be removed from a sequential chain to free capacity for new edits?

## Hypothesis

Removing early (most degraded) edits from the chain will improve survival of remaining edits by freeing weight-space capacity.

## Method

1. **Baseline**: Train 5 sequential edits, measure survival of all 25 facts
2. **GC oldest**: Remove edits 1-2, retrain chain from edit 3, measure survival of remaining 15 facts
3. **GC last only**: Keep only last 2 edits, measure survival of all 25 facts

## Results

```
Strategy                    Survival    Rate    Coverage
---------------------------------------------------------
All 5 edits (baseline)        9/25      36%      25 facts
Remove edits 1-2 (oldest)     8/15      53%      15 facts
Last 2 edits only             5/25      20%      25 facts
```

## Analysis

### 1. GC Improves Per-Edit Survival Rate

Removing old edits increases survival rate for remaining edits:
- Baseline: 9/25 = **36%**
- GC oldest: 8/15 = **53%** (+17% absolute)

This confirms that **old edits consume capacity** that could be used by newer edits.

### 2. But GC Reduces Total Coverage

- Baseline covers **25 facts**
- GC oldest covers only **15 facts**
- Net knowledge: baseline retains 9 facts, GC retains 8 facts

**GC trades breadth for depth.**

### 3. Old Edits Are Truly Gone

Removed edits 1-2 show **0/10 survival** — there is no "ghost memory" of compiled edits after they're removed from the chain. The re-encode step truly erases them.

### 4. Keeping Only Recent Edits Is Worse

Last 2 edits only: 5/25 = **20%** — worse than baseline. This shows:
- Early edits contribute to the model's "edit capacity"
- A longer chain with more edits actually helps (up to a point)
- The degradation is gradual, not catastrophic

## Implications

### When to Use GC

| Scenario | Use GC? | Strategy |
|----------|---------|----------|
| Model at capacity, need new edits | ✅ Yes | Remove oldest |
| Old edits obsolete/deprecated | ✅ Yes | Remove specific |
| Maximize total knowledge | ❌ No | Keep all edits |
| Improve per-edit quality | ⚠️ Maybe | Accept reduced coverage |

### GC Policies

```python
# Policy 1: Remove oldest
wal gc --strategy=oldest --keep=3

# Policy 2: Remove low-survival edits
wal gc --strategy=low-survival --threshold=0.2

# Policy 3: Remove specific edits
wal gc --remove=edit_1,edit_2
```

### Integration with Batch Editing (M229)

GC is more relevant for **sequential editing** than **batch editing**:
- Sequential: Edits overwrite each other → GC frees capacity
- Batch: Facts trained simultaneously → No overwriting → Less need for GC

**Recommendation**: Switch to batch editing (M229) to reduce GC needs.

## Limitations

1. **Small scale**: Only 5 edits tested. Longer chains may show different GC dynamics
2. **No rehearsal**: GC not tested with rehearsal buffer (M228). Rehearsal may change GC calculus
3. **No selective retention**: All facts in an edit are removed together. Selective fact retention within an edit not tested

## Conclusion

> **GC is a viable capacity management tool, but it trades coverage for quality. For maximum knowledge retention, keep all edits and use batch editing instead.**

## Production Recommendation

```
Default:    Keep all edits (maximize retention)
If full:    GC oldest edit, add new edit as batch
If urgent:  GC all but last 3 edits
```
