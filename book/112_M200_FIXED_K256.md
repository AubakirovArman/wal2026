# M200 Fixed — Merge + Re-encode with K=256 (FORWARD BUG FIXED)

**Date:** 2026-04-20  
**Status:** CRITICAL — Previous M200 results invalidated by forward restoration bug

## The Bug

All previous "catastrophic" merge results (M200 K=256, M200b v1-v3 K=1024) were caused by a **code bug in forward restoration**, not structural incompatibility.

After `merge_lora()`, `module.forward` still referenced the LoRA closure, causing **double LoRA application**.

## Analysis

### K=256 is the production sweet spot
- Encode: ~3 minutes (vs ~40 min for K=1024)
- Re-encode cost: +0.079 PPL (vs +0.052 for K=1024)
- Difference: +0.027 PPL — negligible for most use cases

### Two compiled modes

**Fast compile (K=256):**
```
WAL base → LoRA train → merge → re-encode K=256
Time: ~5 min total

```

**Quality compile (K=1024):**
```
WAL base → LoRA train → merge → re-encode K=1024
Time: ~45 min total
Cost: +0.05 PPL
```

## Three Production Modes

### Mode A — Overlay
```
WAL base + LoRA overlay
```
- Flexible runtime editing
- Multiple simultaneous LoRAs
- LoRA forward overhead

### Mode B — Fast Compile (K=256)
```
WAL base → LoRA → merge → re-encode K=256
```
- No runtime overhead
- Single deployable checkpoint
- Fast encoding (~3 min)

### Mode C — Quality Compile (K=1024)
```
WAL base → LoRA → merge → re-encode K=1024
```
- No runtime overhead
- Best quality (+0.05 PPL)
- Slow encoding (~40 min)
