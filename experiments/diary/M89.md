# M89: Streaming Encoder Prototype

## Date
2026-04-23

## Goal
Encode Llama 70B without loading the full model into memory. Process one shard at a time.

## Method
1. Parse `model.safetensors.index.json` to map tensors to shards
2. Download and load one shard (~4.5 GB) at a time
3. Move tensors to GPU, encode with WAL
4. Save WAL representation to disk
5. Free GPU memory
6. Track progress via `progress.json` for resume support

## Results

### Full Model Estimate
| Metric | Value |
|--------|-------|
| Total shards | 30 |
| Total linear weights | 563 |
| Estimated full encode time | **43.6 minutes** |
| Dense size | 135 GB |
| WAL size | 101 GB |
| Savings | 25% |

### Memory Profile
| Metric | Value |
|--------|-------|
| Full model (bf16) | ~140 GB |
| Single shard | ~4.5 GB |
| Streaming encoder peak | **61.7 GB** |
| Memory reduction vs full | 2.3× |

**Note**: Peak GPU memory (61.7 GB) is higher than shard size (4.5 GB) because k-means creates intermediate distance matrices. This fits on H200 (143 GB) but not on consumer GPUs (24 GB).

## Key Insights

1. **Streaming works**: Can encode 70B on single GPU without loading full model.

2. **Resume support works**: `progress.json` tracks completed shards.

3. **GPU memory peaks at 61.7 GB**: k-means distance matrices are the bottleneck. For consumer GPUs, need to encode tensors one at a time (not whole shard at once).

4. **Time penalty vs full model**: 43.6 min (streaming) vs ~30 min (full model). 45% overhead for memory savings.

5. **Output format**: Individual `.wal.pt` per tensor. Easy to parallelize and resume.

## Limitations

1. **Peak GPU memory too high for consumer GPUs** (61.7 GB > 24 GB)
2. **No parallel shard processing** (could use multiple GPUs)
3. **No atom sharing across shards** (each tensor gets independent atoms)
