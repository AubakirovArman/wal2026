# M87: KV-cache Decode Speed Benchmark

## Date
2026-04-23

## Goal
Measure WAL decode throughput for KV-cache and estimate real-world overhead.

## Method
1. Simulate KV-cache: [1, 8, 8192, 128] (one layer, 8K context)
2. Encode K (K=256, C=16) and V (K=64, C=8)
3. Benchmark CPU decode, GPU decode, and precomputed lookup
4. Estimate full-model decode time (80 layers)

## Results

### Decode Speed (8.4M elements per layer)
| Method | K-cache | V-cache |
|--------|---------|---------|
| CPU | 226.5 Mw/s | 260.4 Mw/s |
| GPU | 1,055.7 Mw/s | 5,801.6 Mw/s |
| Precomputed lookup | — | 49,284.7 Mw/s |

**V-cache is 5.5× faster than K-cache on GPU** because K=64 vs K=256. Fewer atoms = faster gather.

**Precomputed lookup is 49 Gw/s** — essentially memory-bandwidth limited. This is the production path.

### Full Model (80 layers, 8K context)
| Format | Size |
|--------|------|
| bf16 | 2.50 GB |
| 12-bit WAL | 1.88 GB |
| 10-bit WAL | 1.56 GB |
| 8-bit WAL | 1.25 GB |

**Estimated decode overhead: 115.7 ms per token** for full 80-layer decode.

### Context Scaling
For 32K context:
| Format | Size | Savings |
|--------|------|---------|
| bf16 | 10.0 GB | — |
| 12-bit | 7.5 GB | 25% |
| 10-bit | 6.3 GB | 37% |
| 8-bit | 5.0 GB | 50% |

## Analysis

### Is 115ms overhead acceptable?
- Llama 70B generates ~20-30 tokens/sec on H200
- Per-token time: 33-50ms
- WAL decode overhead: 115ms
- **Total: 148-165ms → 6-7 tokens/sec**

This is NOT acceptable for real-time generation. The decode overhead dominates.

### How to reduce overhead?
1. **Lazy decode**: Only decode layers that are needed for the current token. Most tokens only need last few layers? No — all layers run every token.
2. **Persistent decoded cache**: Decode once, keep in GPU memory. But then what's the point of compression?
3. **Fused decode**: Decode during attention computation, not before. Custom CUDA kernel.
4. **Quantized decode**: Use INT8 or FP8 for decoded values to reduce memory bandwidth.

### The Real Use Case
KV-cache WAL is NOT for speeding up generation. It's for:
1. **Long-context serving**: Fit more concurrent requests in GPU memory
2. **Offloading**: Compress KV-cache to CPU/disk, decode on demand
3. **Checkpointing**: Save compressed KV-cache state for resuming generation

For real-time inference, the overhead is too high. But for memory-constrained scenarios, 50% memory savings is worth the decode cost.

## Key Insight
KV-cache WAL trades **compute for memory**. It's a memory optimization, not a speed optimization.

## Production Recommendation
- Use 10-bit WAL for K-cache, 8-bit WAL for V-cache
- Decode lazily per layer during forward pass
- Cache decoded layers that are reused
- Accept 100-200ms overhead per token
